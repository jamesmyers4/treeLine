# V2.md — treeline v2 candidate roadmap

_Started after session 20 (v1 complete). This is a holding area for scoped-
but-not-started work — not a commitment, not a schedule. Pick one
initiative at a time. Do not run two in parallel; every session in this
project so far has been single-package-scoped and single-initiative-scoped,
and that discipline is what's kept the codebase bug-free at handoff, past
40+ sessions now._

## How to use this document

1. Pick one initiative below.
2. If its scope isn't already tight (most aren't yet — these are ideas, not
   specs), run a `/grill-me` or `/grill-with-docs` pass on it first, the
   same way diff mode, flow map, GitHub Pages Stage B, the timing signal,
   and AI-proposed assertions were each pressure-tested into a real session
   breakdown before any session prompt got written.
3. Only then draft session prompts, following the same pattern used
   throughout: Step 0 (read real source before assuming anything), exact
   types/signatures, exact test cases, a manual sanity check against real
   data, an explicit Do-not list.
4. Update this file's status line for the initiative as it progresses.
   **Do this promptly — item 3's and item 5's status lines both went stale
   after their sessions completed and had to be corrected retroactively.**
   When an initiative is fully done, fold it into `CONTEXT.md` the same way
   diff mode, flow map, and GitHub Pages Stage B were, and remove it from
   this file.

## Suggested ordering

Cheapest and most-leveraged first — items 2 and 3 both reused data treeline
already captured, no new capture-layer work required, and are both now
done. The remaining items are genuinely bigger swings.

**Calibration note:** visual diffing (formerly item #1 on this list, now
done and folded into `CONTEXT.md`) was estimated at "2-3 sessions" and
actually took 6. GitHub Pages Stage B (item 2) was estimated at "another
session, mostly about picking a rendering approach" and took three
(sessions 34, 35a, 35b), plus a fourth (37) to close a gap found after the
fact. The timing signal (item 4) needed three capture/report sessions
before its first cut was even done. Treat every session estimate on this
file as a floor, not a ceiling.

## 2. Treeline as a GitHub Action + hosted output

**Status:** Complete (sessions 28-37). Stage A (session 28): proven with
two real crawl runs (hgwllc.com, goldenpetbrands.com), including one with
real AI interpretation. Stage B (sessions 34-35b): `@treeline/pages`
renders each run to static HTML and publishes it to the `gh-pages` branch,
opt-in via a `publish_to_pages` input, proven with real published runs. The
root-URL gap flagged after Stage B (bare Pages URL 404ing, only `/runs/`
working) was closed in session 37 — see `CONTEXT.md`'s "V2 additions" for
the full architecture writeup, the boolean-input bug found and fixed along
the way, and the real GPB judgment call that shaped the opt-in default.
Session 43 (not itself a roadmap item, but directly tied to this one)
hardened the published HTML against content-injection from crawled data —
see `CONTEXT.md` for detail.

**What:** package `crawl` + `diff` behind a `workflow_dispatch` trigger
with a `url` input field (this is the free "plug in a URL" UI — GitHub
renders it as an actual form, no custom frontend needed), secrets-based
API key, and `actions/upload-artifact` for the output directory, plus
auto-publish of rendered reports to GitHub Pages after each run (shiki for
`.ts` file highlighting, markdown-it for `.md` reports) so there's a
shareable link, not just a zip to download.

**Why:** `crawl` + `diff --fail-on-regression` already _is_ a full CI
regression gate — every competitor researched pitches "runs on every
deploy" as the headline feature, and treeline can already do that today.
This is packaging work, not new capability.

**Watch for:** Playwright browser installation is slow without caching —
budget time for getting `playwright install --with-deps` cached properly
in the workflow. GitHub-hosted runner IPs are datacenter ranges; expect
more bot-detection friction than crawling from a home connection
(goldenpetbrands.com's Cloudflare challenge is the known example).
GitHub-hosted runners need Xvfb, since the default (non-stealth) capture
path launches headed with no CLI override. This repo is public, so Actions
logs/artifacts from any crawl run are visible to anyone (the API key
itself is masked, but crawled content is not), and publishing a real
third-party target to `gh-pages` is a deliberate choice, not a default —
see the GPB judgment call in `CONTEXT.md`. The Action deliberately doesn't
expose `--stealth`; stealth-mode testing stays local-only for now.

## 3. Coverage-gap reporting

**Status:** Complete (session 38). `coverage-report.md`, wired into
`treeline crawl` as the sixth automatic report. Four sections: zero-
coverage pages, high-skip pages (>50% of a page's elements skipped —
relative, not absolute), forms without a corresponding generated test
(found to be a currently-always-true fact about `generateSpec`, stated
plainly rather than framed as a per-form finding it isn't), and unresolved
`hard-pages/` entries. No new capture or persistence work, as originally
scoped — see `CONTEXT.md`'s "V2 additions" for the full writeup, including
a bundled, unrelated fix in the same session (a real flow-map table
overflow bug on long unbroken URLs).

**What:** a summary view of what _isn't_ covered — pages with zero
generated POM coverage, pages with unusually high skipped-element counts,
forms with no corresponding generated test, unresolved `hard-pages/`
entries.

**Why:** Rainforest QA sells exactly this as a distinct feature
("visibility into your actual coverage," not just individual test
generation). Treeline already tracked every input this needed.

## 4. Timing / flakiness signal

**Status:** Complete (sessions 39-41, 44) — page-load timing capture,
single-run reporting, and diff-mode regression detection all done.
Network-request and element-appearance-latency diffing are new,
separately-scoped follow-up ideas (see "New follow-ups" below), not part
of this item.

**What:** surface pages likely to produce flaky tests — slow-loading
pages, elements with unusually long appearance latency, slow network
requests.

**Why:** industry research (QA Wolf's public breakdown) puts timing-related
failures at roughly 30% of real test failures — larger than selector
issues, which is what treeline's diff mode currently addresses. Treeline
had zero signal on this category before this arc.

**Done (sessions 39-41):** despite this item's original description
claiming it could reuse "capture timing data ... that already exists,"
that claim turned out to be false on inspection — no timestamps or
durations existed anywhere in `PageState`/`NetworkEntry` before this arc.
New capture work in `packages/acquire` (sessions 39-40): `pageLoadMs`,
`NetworkEntry.durationMs`, and `DomInteractiveElement.appearedAtMs`
(narrowed to only mean something for elements rendered in after initial
load, via a `MutationObserver`). `packages/output/src/timing-report.ts`
(session 41), wired in as the seventh automatic report
(`timing-report.md`), three sections each with an empirically-derived
threshold (2500ms page load, 500ms network request — both solid; 2500ms
appearance latency — thinner basis, real crawls turned up too few natural
samples, so it's anchored to Playwright's default `expect()` timeout
instead, stated explicitly rather than presented with false confidence).
Full detail, including the real thresholds' derivation data, in
`CONTEXT.md`'s "V2 additions".

**Done (session 44):** diff-mode timing regression detection.
`packages/core/src/timing-diff.ts`, page-load timing only, matched on
normalized URL. `TIMING_NOISE_THRESHOLD_PERCENT = 50`, derived from real
repeated crawls (local fixture noise +/-6.4%, real external multi-page
site +/-17%, real external single-page site normally +/-15% but one
observed +126% cold-connection outlier — documented as a known,
unresolved false-positive source). Wired into `CrawlDiff.timingChanges`,
a new `## Page Load Timing Changes` section in `diff-report.md`, and
`--fail-on-regression` confirmed (by test and by a real crawl) to never
trip on a timing-only regression — report-only, same guarantee visual
changes already had. Full derivation data in `CONTEXT.md`'s "V2
additions".

**New follow-ups (not part of this item's original scope):**
network-request timing diffing and element-appearance-latency diffing
across two crawls — same category of feature, deliberately not built this
session to keep the cut narrow, same pattern as every other feature in
this project's history.

## 5. AI-proposed test assertions (beyond skeleton specs)

**Status:** First cut complete (session 42) — forms only, deliberately
narrowed.

**What:** extend `packages/interpret`'s existing `PageInterpretation`
(`pageType`, `purpose`, `keyDataEntities`) to propose an actual
assertion-rich test scenario for a page's evident purpose, not just an
empty-ish spec skeleton — e.g. a page interpreted as a checkout form gets
a proposed test that fills it and asserts a success state.

**Why:** this is the most "out of the box AI" item on this list, and it
builds on data the interpretation layer already computes rather than
requiring new capture work.

**Done (session 42), scoped tightly after a real `/grill-me` pass before
any code was written — this was the first item to touch generated test
_logic_, not just structure, and needed a human-review posture worked out
first:**

- `StoredInterpretation.proposedAssertion: ProposedAssertion | null`,
  attempted only for pages with a captured form.
- Every proposal renders to a **separate** `*.proposed.spec.ts` file,
  **never merged into the trusted spec**.
- Every proposed test is wrapped in **`test.skip(...)`** — real, complete,
  editable code, never runnable by default. Verified for real by actually
  running the generated file through `npx playwright test` and confirming
  it reports skipped, not executed.
- Fill values are deliberately, obviously synthetic (`"Test User"`,
  `"test@example.com"`) — a second safety layer independent of
  `test.skip`.
- Treeline itself never fills or submits a real form during generation —
  stays entirely passive, same as everywhere else in this codebase.
- The success assertion is explicitly labeled an unverified guess, in the
  generated file itself — treeline has never observed real
  post-submission behavior for any page it's crawled.
- A real bug was caught by the mandated manual check: the model's own
  freeform guess at a field's `accessibleName` was originally used as the
  matching key back to real captured field data, which silently broke on
  genuinely-unlabeled real-world inputs. Fixed — see `CLAUDE.md`'s
  gotchas for the general principle this established (never use a
  model's freeform output as a lookup key against structured data).
- Session 43's audit separately confirmed all the dynamic values spliced
  into generated proposed specs are safe by construction
  (`JSON.stringify`), except two that needed a different technique
  (`toSafeComment`) — fixed in that session, detailed in `CONTEXT.md`.

**Deliberately not built in this first cut, real future extensions:**
scenarios beyond form-fill-and-submit; a lightweight way to "promote" a
reviewed-good proposal into the trusted spec (plain copy-paste may be
entirely sufficient — no tooling has been judged necessary yet); a summary
report of how many proposals exist across a crawl, similar in spirit to
`coverage-report.md`.

## 6. Multi-step flow test generation

**Status:** not started (this is the existing Phase 2 backlog item from
`CONTEXT.md` — interaction-reachable page discovery — reframed here as
what it actually unlocks)

**What:** generate tests that span multiple pages as one flow ("add to
cart → checkout → confirm"), not one POM per page in isolation.

**Why:** this is the single biggest structural gap between treeline and
the autonomous-agent platforms researched (BlinqIO, mabl, QA Wolf), all of
which center on flows, not pages.

**Rough shape:** the biggest item here by a wide margin — depends on
interaction-reachable discovery (crawling states only reachable via
click/form-submit), which is a real new capability, not a fix or an
extension. Don't start this until there's real appetite for a diff-mode-
sized multi-session arc.

## 7. Hosted run + output viewer (staged)

**Status:** Stage A and Stage B done (sessions 28-37); see item 2 above
for the full writeup rather than duplicating it here. Stage C not started.

**What:** three increasingly ambitious stages, not one project:

- **Stage A** and **Stage B** — both done; see item 2. Still no backend,
  no server to maintain — a `workflow_dispatch` trigger plus a static-HTML
  publish to `gh-pages`.
- **Stage C** — a real custom web frontend: URL input triggers a run
  (via the GitHub Actions REST API, polling for the artifact — Playwright
  doesn't run well on serverless, so this still leans on Actions for
  compute rather than hosting a browser environment directly), plus a
  Monaco-based file browser for the output. Monaco's built-in diff editor
  is a strong fit specifically for `diff-report.md` — side-by-side,
  syntax-highlighted before/after.

**Why:** turns treeline from "a CLI you run" into something a non-CLI
person (or an interviewer) could actually try.

**Rough shape:** Stage C is its own multi-session arc — needs a real
scoping pass on hosting/triggering architecture before session prompts
get written, don't treat it as a quick add-on.

## 8. Auto-file bugs to an external tracker (Jira or similar)

**Status:** idea captured, not scoped

**What (as raised, not yet resolved):** wire treeline up to automatically
create a ticket in an external tracker when something's found — Jira as
the concrete example, but scoped generically enough to swap in another
tool later. Probably toggleable, since it's not obviously something you'd
want running all the time.

**Open question the idea itself raised — worth returning to, not resolving
now:** what should actually trigger a ticket. First framing was "any
finding," immediately self-questioned. Sharper candidate that came up
mid-thought: only file when a finding is a genuine defect in the site/app
itself, not a problem with treeline's own generated test code — which
would mean treeline needs some way to tell "real bug" apart from "my test
is stale/wrong" before it could responsibly decide to file anything. That
distinction is the actual crux of whether this is worth building at all.

**Rough shape:** unscoped. Needs a real conversation, likely its own
`/grill-me` pass, before this even gets a rough estimate — the
trigger-condition question above has to get answered first.

## Known strength worth remembering while prioritizing

Several paid competitors researched store tests in proprietary formats
with no export path (mabl, notably). Treeline generates plain, ownable
TypeScript with no vendor lock-in and no per-seat pricing. Nothing on this
list should compromise that — if any future initiative would mean
generated output stops being plain, readable, portable code, that's a
reason to reconsider the approach, not just a tradeoff to accept. This has
held through every V2 item so far, including the two that touch AI-
generated code most directly (items 4 and 5) — proposed assertions are
still plain, readable `.ts`, just gated behind `test.skip` rather than
hidden behind any proprietary mechanism.
