# V2.md — treeline v2 candidate roadmap

_Started after session 20 (v1 complete). This is a holding area for scoped-
but-not-started work — not a commitment, not a schedule. Pick one
initiative at a time. Do not run two in parallel; every session in this
project so far has been single-package-scoped and single-initiative-scoped,
and that discipline is what's kept 20 sessions bug-free at handoff._

## How to use this document

1. Pick one initiative below.
2. If its scope isn't already tight (most aren't yet — these are ideas, not
   specs), run a `/grill-me` or `/grill-with-docs` pass on it first, the
   same way diff mode and flow map were each pressure-tested into a real
   session breakdown before any session prompt got written.
3. Only then draft session prompts, following the same pattern used
   throughout v1: Step 0 (read real source before assuming anything),
   exact types/signatures, exact test cases, a manual sanity check against
   a real site, an explicit Do-not list.
4. Update this file's status line for the initiative as it progresses.
   When an initiative is fully done, fold it into `CONTEXT.md` the same
   way diff mode and flow map were, and remove it from this file.

## Suggested ordering

Cheapest and most-leveraged first — each of the first three reuses data
treeline already captures, no new capture-layer work required. The last
two are genuinely bigger swings and shouldn't be started until there's
appetite for a multi-session arc like diff mode was.

---

## 1. Visual diffing in `treeline diff`

**Status:** not started

**What:** `PageState` has captured a screenshot on every page, every crawl,
since session 1 — nothing has ever used it. Add a visual comparison step to
diff mode: pixel-diff (or better) between a page's baseline and current
screenshot, surfaced as its own section in `diff-report.md` alongside the
existing selector-regression section.

**Why:** This is the single most-leveraged idea on this list — the data
already exists, diff mode already has the right shape (compare two runs,
surface what changed). It also puts treeline in the same conversation as
an entire funded product category (Applitools, Percy, Chromatic) at a
fraction of the build cost, since treeline doesn't need cross-browser
rendering or a hosted review workflow to be useful here.

**Rough shape:** probably 2-3 sessions — comparison logic in
`packages/core` (or a new small package if it doesn't fit cleanly
alongside the existing selector diff), report rendering in
`packages/output`, CLI wiring. A stretch addition once the basic version
works: a multimodal Claude call that looks at a before/after pair and
describes _what_ changed in plain language — genuinely AI-native, not
something the pixel-diff competitors do out of the box.

**Watch for:** false positives from font rendering / anti-aliasing
differences between crawl runs — every visual-diff competitor researched
treats this as the central hard problem, not an edge case. Don't ship a
naive pixel-diff without a noise threshold, or the report will be
unusable on real sites.

## 2. Treeline as a GitHub Action + hosted output

**Status:** not started

**What:** package `crawl` + `diff` behind a `workflow_dispatch` trigger
with a `url` input field (this is the free "plug in a URL" UI — GitHub
renders it as an actual form, no custom frontend needed), secrets-based
API key, and `actions/upload-artifact` for the output directory. Stage 2:
auto-publish reports to GitHub Pages after each run (Shiki for `.ts` file
highlighting, native markdown rendering for `.md` reports) so there's a
shareable link, not just a zip to download.

**Why:** `crawl` + `diff --fail-on-regression` already _is_ a full CI
regression gate — every competitor researched pitches "runs on every
deploy" as the headline feature, and treeline can already do that today.
This is packaging work, not new capability.

**Rough shape:** Stage 1 (workflow + artifacts) is genuinely small, maybe
one session. Stage 2 (Pages auto-publish) is another session, mostly
about picking a markdown-to-HTML approach and wiring Shiki into a static
build step.

**Watch for:** Playwright browser installation is slow without caching —
budget time for getting `playwright install --with-deps` cached properly
in the workflow. Also: GitHub-hosted runner IPs are datacenter ranges;
expect more bot-detection friction than crawling from a home connection
(goldenpetbrands.com's Cloudflare challenge is the known example).

## 3. Coverage-gap reporting

**Status:** not started

**What:** a summary view of what _isn't_ covered — pages with zero
generated POM coverage, pages with unusually high skipped-element counts,
forms with no corresponding generated test, unresolved `hard-pages/`
entries. Could be its own report or a new section in the atlas.

**Why:** Rainforest QA sells exactly this as a distinct feature
("visibility into your actual coverage," not just individual test
generation). Treeline already tracks every input this needs — skipped
elements, hard-pages, POM generation counts — none of it is currently
surfaced as a single view.

**Rough shape:** likely one `packages/output` session — no new capture or
persistence work, this is entirely a rendering/aggregation task over data
that already exists.

---

## 4. Timing / flakiness signal

**Status:** not started

**What:** surface pages likely to produce flaky tests — slow-loading
pages, elements with unusually long appearance latency — using capture
timing data and network log data that already exists.

**Why:** industry research (QA Wolf's public breakdown) puts timing-related
failures at roughly 30% of real test failures — larger than selector
issues, which is what treeline's diff mode currently addresses. Treeline
has zero signal on this category today.

**Rough shape:** needs a scoping pass before session prompts — unlike
items 1-3, this may need new capture-layer instrumentation (`packages/
acquire`), not just new reporting over existing data. Run a `/grill-me`
pass on this one specifically before writing session prompts.

---

## 5. AI-proposed test assertions (beyond skeleton specs)

**Status:** not started

**What:** extend `packages/interpret`'s existing `PageInterpretation`
(`pageType`, `purpose`, `keyDataEntities`) to propose an actual
assertion-rich test scenario for a page's evident purpose, not just an
empty-ish spec skeleton — e.g. a page interpreted as a checkout form gets
a proposed test that fills it and asserts a success state.

**Why:** this is the most "out of the box AI" item on this list, and it
builds on data the interpretation layer already computes rather than
requiring new capture work.

**Rough shape:** needs real scoping — this is the first item that touches
generated test _logic_ rather than structure, and generated logic needs a
human-review posture (never auto-commit an AI-proposed assertion as
ground truth) thought through before any code gets written. `/grill-me`
this one before anything else.

---

## 6. Multi-step flow test generation

**Status:** not started (this is the existing Phase 2 backlog item from
`CONTEXT.md` — interaction-reachable page discovery — reframed here as
what it actually unlocks)

**What:** generate tests that span multiple pages as one flow ("add to
cart → checkout → confirm"), not one POM per page in isolation.

**Why:** this is the single biggest structural gap between treeline and
the autonomous-agent platforms researched (BlinqIO, mabl, QA Wolf), all of
which center on flows, not pages.

**Rough shape:** the biggest item here by a wide margin — depends on
interaction-reachable discovery (crawling states only reachable via
click/form-submit), which is a real new capability, not a fix or an
extension. Don't start this until there's real appetite for a diff-mode-
sized multi-session arc.

---

## 7. Hosted run + output viewer (staged)

**Status:** not started

**What:** three increasingly ambitious stages, not one project:

- **Stage A** — GitHub Actions `workflow_dispatch` with a `url` input +
  `actions/upload-artifact`. (Overlaps with item 2, Stage 1 — do these
  together.)
- **Stage B** — auto-publish reports to GitHub Pages after each run.
  Shiki for `.ts` syntax highlighting, native markdown rendering for
  reports. Still no backend, no server to maintain.
- **Stage C** — a real custom web frontend: URL input triggers a run
  (via the GitHub Actions REST API, polling for the artifact — Playwright
  doesn't run well on serverless, so this still leans on Actions for
  compute rather than hosting a browser environment directly), plus a
  Monaco-based file browser for the output. Monaco's built-in diff editor
  is a strong fit specifically for `diff-report.md` — side-by-side,
  syntax-highlighted before/after.

**Why:** turns treeline from "a CLI you run" into something a non-CLI
person (or an interviewer) could actually try.

**Rough shape:** Stage A/B are small, covered under item 2. Stage C is its
own multi-session arc — needs a real scoping pass on hosting/triggering
architecture before session prompts get written, don't treat it as a
quick add-on.

---

## Known strength worth remembering while prioritizing

Several paid competitors researched store tests in proprietary formats
with no export path (mabl, notably). Treeline generates plain, ownable
TypeScript with no vendor lock-in and no per-seat pricing. Nothing on this
list should compromise that — if any future initiative would mean
generated output stops being plain, readable, portable code, that's a
reason to reconsider the approach, not just a tradeoff to accept.
